# 用 LLM Coding Agent 解决复杂任务（WIP）

成文于2026年3月，仅代表此刻情况。

## 什么是 “复杂任务”

- 丰富的环境：完成任务需要与多种系统、数据、工具交互
- 隐含上下文：许多信息，如数据格式、工具用法、常见坑点，没有成文文档可供参考
- 长逻辑链条：解决任务需要大量不同层次的推理
- 开放的目标：没有一个明确的、唯一的正确答案。甚至不一定有明确的成功标准。

### 首先加钱

处理复杂任务当然需要最强力的帮手。
如果你还没有使用业界最领先的模型之一，务必先想办法试试。

## LLM Agent 特点回顾

优点：

- 极大量的预训练知识
- 可观的自然语言读写能力
- 可观的代码读写能力
- 低运行成本（相较于人类）
- 分解任务并自主运行一定时间的能力
- 开新session就可以轻松遗忘近期记忆（人类做不到）

缺点：

- 没有在提示词之外获取目标的能力
- 上下文窗口有限
- 没有原生的可变长期记忆

优点部分已经基本满足我们的需求了。即使有不满足的地方，我们也较难干预。

因此在使用 LLM Agent 解决复杂任务时，核心在于如何规避缺点。

### 模型能力增强会改变互动方式吗？

完全可能，但当模型能力增强的时候，我们也会期望用它完成更复杂的任务。

本文尽量介绍一些与当前模型能力无关的内容。

## 人的价值

### 背锅（划掉） 责任

[积极主动](https://confluence.zding.fund/pages/viewpage.action?pageId=314033123)

每个人是自己 Agent 的第一责任人。
Agent 的成果，无论好坏，最终还是由物理世界中的人来承受。

Agent 并非包治百病的灵丹妙药，完全可能出现各种问题。
与 Agent 的互动中缺少深度投入，是一种智性懒惰。

在使用 Agent 时，应当有意识地核验 Agent 的产出，并寻求改善 Agent 表现的方法。

### 目标

[要事第一](https://confluence.zding.fund/pages/viewpage.action?pageId=314033123)

价值理性：什么是好的？

工具理性：什么是有用的？什么是好用的？

NP 问题的启示：让 Agent 提供证据证明它的工作成果是对的，我们检查起来远比深挖工作过程要容易。

### 合理授权

对相信 Agent 能自行完成的工作，可以少干预过程。

但如果认为复杂程度超出了 Agent 自行完成的阈值，则需要合理干预以掌控进展方向。
Agent 工作过程中的关键结论与决策，自己最好能够把关，至少也要知情。

### 关爱大脑健康

Agent 会把人从执行者变成监督者，如果没有刻意设计工作方式，很容易陷入长时间盯屏、频繁切换、不断追问细节的脑力透支状态。和 Agent 协作时，目标不是把所有思考都塞进自己脑子里，而是把“需要记住什么、现在做到哪一步、接下来验证什么”尽量外化。

更具体地说，可以把复杂任务拆成几个明确阶段：先约定目标与验收标准，再让 Agent 自主推进一段时间，只在关键节点查看证据和结论；把重要背景、决定、待办写进文档或 checklist，减少反复回忆上下文；当发现结果跑偏时，优先补充上下文，而不是靠连续下指令硬拧方向。这样既能降低自己的认知负担，也更容易看清问题究竟出在目标、信息还是执行。

如果任务持续时间很长，还要主动安排“换脑子”的机制：限制单次盯盘时长，遇到卡点先停下来整理现状，再决定是继续、回滚还是换方案。人要负责判断价值与风险，Agent 负责放大执行效率；把二者边界划清，才能既用好工具，也不把自己耗干。

## 长期记忆

### 新同事入职

Agent 每天都是第一天工作【哭】。
对他好一点，起码按对待新同事的标准来准备入职文档。

### 从小例子开始

接触一个新项目时，一下太多内容很难吸收。

可以让 Agent 在一些小规模例子上先跑跑看，并将经验沉淀到文档中，作为一个好的起点。

参考：[沙盒演练](https://confluence.zding.fund/pages/viewpage.action?pageId=314026272)

### 持久化

session 里的信息在关闭之后就消失了，这不利于记忆的积累。
尽量将你的输入写入文件，也将 Agent 值得保存的输出写入文件。

长期来看大量的文档形成一种资产，是 Agent 后续工作的知识库。

#### 典型场景1【别让我说第二次】：

1. Agent 在某个步骤犯了一个常见错误，你发现了。
2. 这时就应该把这个错误、出错的上下文与原因以及正确的做法写入文档了（可以指示 Agent 自己写，人只负责审阅）

#### 典型场景2【前期文档】：

让 Agent 将它的调研思路、设计思路写入文档。
当我们审阅文档中的思路不符合预期时，可以补充信息，要求 Agent 修正。
写入文档此时方便我们查看 diff

在调研、设计完成后，Agent 的上下文可能已经比较满了。
如果反复修改次数多，里面可能还有一些错误思路。
此时文档就是项目的 checkpoint，我们可以基于文档重新开一个新的 session 来继续后续的工作。

#### 典型场景3【进展管理】：

前期工作完成后，就要开始正式的开发了。
在开发过程中，Agent 可能会遇到各种各样的问题，犯各种各样的错误。
如果只通过对话干预，效率比较低，而且控制力度不足。

如果让 Agent 一开始就写一份包含工作项的进展文档，并在后续的工作中持续更新这个文档，那么我们就可以大胆丢掉先前的 session，以全新的干净 context 来完成后续工作了。

### 软硬结合

对 Agent 而言，代码类似可编程硬件，适合处理封闭任务，自然语言则更为灵活柔软，适合处理开放任务。
Agent 可以在无人干预的情况下用自然语言思考，并组合若干命令来完成任务，这很智能。
但这并不表示这就是完成任务的最优方式。

对于硬性封闭知识，最好的方法就是写硬性的代码。

#### 典型场景：某个 API 应该按特定的方式调用

自然语言方案：在提示词文件中某处记录

- 人和 Agent 都可自主维护
- 但自然语言描述不一定能保证约束 Agent 的行为
- 后续再错不一定能保证改好，只能补充提示词

编程方案：提供一个调用该 API 的库，封装相关细节

- 人和 Agent 仍然都可以自主维护
- 保证行为符合代码逻辑
- 后续再错是明确的 库bug 或者 调用bug，清晰好改
- 上下文占用小，没出错时不产生复杂思考

反思：人类工作中也常存在欠使用工具的情况。这可能是因为长期以来软件工具的开发成本较高，导致人们习惯于直接用手工操作来完成偶发或者不够封闭的任务。但 Agent 大幅降低了工具的开发成本，因此我们需要重新审视原来由于成本问题而没有工具协助的场景。
