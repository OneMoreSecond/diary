# 用 LLM Coding Agent 解决复杂任务

成文于2026年3月，仅代表此刻情况。

## 什么是 “复杂任务”

- 丰富的环境：完成任务需要与多种系统、数据、工具交互
- 隐含上下文：许多信息，如数据格式、工具用法、常见坑点，没有成文文档可供参考
- 长逻辑链条：解决任务需要大量不同层次的推理
- 开放的目标：没有一个明确的、唯一的正确答案。甚至不一定有明确的成功标准。

### 首先加钱

处理复杂任务需要最强力的帮手。
如果还没有在使用业界最领先的模型之一，务必先试试。

## LLM Agent 特点回顾

优点：

- 极大量的预训练知识
- 可观的自然语言阅读能力
- 可观的代码阅读能力
- 一定的代码生成能力
- 一定的代码写作能力
- 低运行成本（相较于人类）
- 分解任务并自主运行一定时间的能力
- 开新session就可以轻松遗忘近期记忆（人类做不到）

缺点：

- 没有在提示词之外获取目标的能力
- 上下文窗口有限
- 没有原生的可变长期记忆

优点部分已经基本满足我们的需求了。即使有不满足的地方，我们也较难干预。

因此在使用 LLM Agent 解决复杂任务时，核心在于如何规避缺点。

### 模型能力增强会改变互动方式吗？

完全可能，但当模型能力增强的时候，我们也会期望用它完成更复杂的任务。

本文尽量介绍一些与当前模型能力无关的内容。

## 人的价值

### 责任

每个人是自己 Agent 的第一责任人。
Agent 的成果，无论好坏，最终还是由物理世界中的人来承受。

Agent 并非包治百病的灵丹妙药，完全可能出现各种问题。
与 Agent 的互动中放弃深度思考，是一种**智性懒惰**。

### 目标

价值理性：什么是好的？

工具理性：什么是有用的？什么是好用的？

NP 问题的启示：让 Agent 提供证据证明它的工作成果是对的，我们检查起来远比深挖工作过程要容易。

### 合理授权

对相信 Agent 能自行完成的工作，可以少干预过程。

但如果认为复杂程度超出了 Agent 自行完成的阈值，则需要合理干预以掌控进展方向。
Agent 工作过程中的关键设计、决策与结论，自己最好能够把关，至少也要知情。

### 关爱大脑健康

同时操控多个 Coding Agent 时，大脑并不是在“写代码”，而是在做更高一层的控制：拆任务、看结果。
人脑的主要负担从局部逻辑推理，转向调度、监控和决策，而这些负担主要压在前额叶（工作记忆 + 任务切换）上。真正消耗脑力的往往不是某一个技术问题本身，而是反复进行 context switch、结论验证，以及一连串细碎但密集的 micro-decision。

一个有效的优化方向，是把调度尽量外部化：把任务步骤、agent分派、验收标准和当前状态写出来，不要把它们全塞在脑子里。否则，前额叶就会被迫充当 in-memory scheduler；前期还能顶住，时间一长就容易出现警觉下降、误判结果，甚至觉得“盯 Agent 比自己写还累”。

### 工欲善其事，必先利其器

想要高效的工作，需要趁手的工具。

工具既可以是软性的提示词文档，也可以是硬性的程序：

- 一份描述开源工具用法的 Skill 文档
- 一个自行改动过的工具 fork

### Context over control

满怀期待向 Agent 交办一项任务，却发现结果方向有偏差。此时不妨尝试站在 Agent 的角度，了解 Agent 所获取的信息，以及这样做的原因。想想缺了什么信息，将它补充到上下文中。

### 以终为始

一开始就要告诉 Agent 任务结束标准是什么。自然语言描述的模糊标准也比没有强。

例如：

- 跑通某某测试才结束

## 长期记忆

### 新同事入职

Agent 每天都是第一天工作【哭】。
对他好一点，起码按对待新同事的标准来准备入职文档。

### 从小例子开始

接触一个新项目时，一下太多内容很难吸收。

可以让 Agent 在一些小规模例子上先跑跑看，并将经验沉淀到文档中，作为一个好的起点。

### 持久化

session 里的信息在关闭之后就与项目分离了，这不利于记忆的积累。
尽量将人类输入写入文件，也将 Agent 值得保存的输出写入文件。
此时文档就是任务的 living summary，我们可以随时将任务信息转交给其他人或者Agent。

长期来看大量的文档形成一种资产，是 Agent 后续工作的知识库。

#### 典型场景【前期文档】

人类简要介绍任务背景信息及目标后，可以让 Agent 将事实调研、边界假设、关键决策、设计思路写入文档。
文档形成后，人类可以查看其内容是否与任务预期对齐。
写入文档此时方便我们查看 diff 验证改动是否到位。

#### 典型场景【进展管理】

在开发过程中，Agent 可能会遇到各种各样的问题，犯各种各样的错误。
如果只通过对话干预，效率比较低，而且控制力度不足。

如果让 Agent 写一份包含工作项的进展文档，并在后续的工作中持续更新这个文档，那么我们就可以通过修改进展文档并新建 session，修正错误的历史记忆。

### 记忆层次

在一个任务中，并非所有历史信息都是有用的。
过多的信息会使任务失焦。
对大模型来说，唤起记忆只要读文件就行了；但遗忘只有在新建 session 时才能发生。
因此我们需要管理好记忆金字塔，使模型按需唤起记忆。
记忆的入口应当尽量简略。

示例：

- Skills
- 项目内的多级记忆目录，带索引

#### 记忆更新由 Reward 驱动

何时改动记忆呢？
如果模型能在当前记忆情况下完成好任务，那么就不需要改。
比较实用的标准是：由Agent的错误驱动。
人类观察到错误，或者Agent自行发现错误时，都是更新记忆的良机。

### 软硬结合

对 Agent 而言，代码类似可编程硬件，适合处理封闭任务，自然语言则更为灵活柔软，适合处理开放任务。
Agent 可以在无人干预的情况下用自然语言思考，并组合若干命令来完成任务，这很智能。
但这并不表示这就是完成任务的最优方式。

对于硬性封闭知识，最好的方法就是将临时的软思考过程转化为硬性的代码，用精简的接口来封装复杂的逻辑。

反思：人类工作中也常存在欠使用工具的情况。
这可能是因为长期以来软件工具的开发成本较高，导致人们倾向于用软思考过程操作来完成偶发或者不够封闭的任务。
但 Agent 大幅降低了工具的开发成本，因此我们需要重新审视原来由于成本问题而没有工具协助的场景。
