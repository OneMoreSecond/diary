# 用 LLM Coding Agent 解决复杂任务（WIP）

成文于2026年3月，仅代表此刻情况。

## 什么是 “复杂任务”

- 丰富的环境：完成任务需要与多种系统、数据、工具交互
- 隐含上下文：许多信息，如数据格式、工具用法、常见坑点，没有成文文档可供参考
- 长逻辑链条：解决任务需要大量不同层次的推理
- 开放的目标：没有一个明确的、唯一的正确答案。甚至不一定有明确的成功标准。

#### 首先加钱

处理复杂任务当然需要最强力的帮手。
如果你还没有使用业界最领先的模型之一，务必先想办法试试。

## LLM Agent 特点回顾

优点：

- 极大量的预训练知识
- 可观的自然语言读写能力
- 可观的代码读写能力
- 低运行成本（相较于人类）
- 分解任务并自主运行一定时间的能力
- 开新session就可以轻松遗忘近期记忆（人类做不到）

缺点：

- 没有在提示词之外获取目标的能力
- 上下文窗口有限
- 没有原生的可变长期记忆

优点部分已经基本满足我们的需求了。即使有不满足的地方，我们也较难干预。

因此在使用 LLM Agent 解决复杂任务时，核心在于如何规避缺点。

## 人的价值

#### 背锅（划掉） 责任

[积极主动](https://confluence.zding.fund/pages/viewpage.action?pageId=314033123)

每个人是自己 Agent 的第一责任人。
Agent 的成果，无论好坏，最终还是由物理世界中的人来承受。

Agent 并非包治百病的灵丹妙药，完全可能出现各种问题。
在使用 Agent 时，应当时时主动寻求改善 Agent 表现的方法。

#### 目标

[要事第一](https://confluence.zding.fund/pages/viewpage.action?pageId=314033123)

价值理性：什么是好的？

工具理性：什么是有用的？什么是好用的？

判断题思维：“判断Agent的提议好坏” 远比 “给Agent一个好提议” 要容易。

#### 合理授权

对相信 Agent 能自行完成的工作，可以少干预过程。

但如果认为复杂程度超出了 Agent 自行完成的阈值，则需要合理干预以掌控进展方向。
Agent 工作过程中的关键结论与决策，自己最好能够把关，至少也要知情。

#### Context over control

[知彼解己](https://confluence.zding.fund/pages/viewpage.action?pageId=314033123)

当你满怀期待的向 Agent 交办一项任务，去发现结果方向有偏差。常见的反应有什么？

- 要求 Agent 再想想，深刻领会指令精神
- 发出新的指令，要求 Agent 按你说的改
- 不换思想就换人，换一个模型来试试

不妨尝试站在 Agent 的角度，了解 Agent 所获取的信息，以及这样做的原因。
想想缺了什么信息，将它沉淀到提示词文件中，使用这种方法来改变 Agent 的行为。

## 长期记忆

#### 新同事入职

Agent 每天都是第一天工作【哭】。
对他好一点，起码按对待新同事的标准来准备入职文档。

#### 从小例子开始

接触一个新项目时，一下太多内容很难吸收。

可以让 Agent 在一些小规模例子上先跑跑看，并将经验沉淀到文档中，作为一个好的起点。

#### 持久化

session 里的信息在关闭之后就消失了，这不利于记忆的积累。
尽量将你的输入写入文件，也将 Agent 值得保存的输出写入文件。

##### 典型场景1【别让我说第二次】：

1. Agent 在某个步骤犯了一个常见错误，你发现了并告诉了它。
2. 过了一段时间，你又发现它又犯了同样的错误。
3. 这时就应该考虑把这个错误以及正确的做法写入文档了。
